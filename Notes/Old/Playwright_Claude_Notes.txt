Lecture 14 — Browser Factory Design Pattern


What is Browser Factory?


Browser Factory is a design pattern used to create and manage browser instances from one single place.
Instead of writing browser launch code again and again in every test file, you write it once in a BrowserFactory class and reuse it everywhere.
Think of it like a remote control — one remote can control any TV (Chromium, Firefox, WebKit). You don't buy a new remote for every TV.



Why Do We Use It?


Avoids code duplication — No need to write browser launch code in every test.
Easy maintenance — If you want to change something (e.g., headless mode), you change it in one place only.
Supports multiple browsers — Launch Chromium, Chrome, Firefox, Edge, or WebKit from one method.
Clean architecture — Follows OOP principles like Encapsulation.



Key Points to Remember


⚡ Firefox and WebKit do NOT support --start-maximized argument. Only Chromium-based browsers (Chrome, Edge, Chromium) support it.
⚡ For Firefox/WebKit fullscreen, you must set viewport dimensions equal to the screen resolution manually.
⚡ Encapsulation is used by declaring playwright and browser as private static — they are hidden from outside and managed internally.
⚡ Always call playwright.close() after tests finish. This closes all browsers and releases memory.
🐛 Common Mistake in the original code: There was an extra orphaned browser launch line that was never assigned or closed — always make sure every launched browser is assigned to a variable and closed properly.


java// ❌ Wrong — This launches a browser nobody uses. Wastes memory.
playwright.chromium().launch(new BrowserType.LaunchOptions().setHeadless(false).setSlowMo(1000));

// ✅ Correct — Assign launch result to a variable if needed, or remove this line.


Memory Trick: Think of BrowserFactory as a car showroom — you tell it what car (browser) you want, it hands it to you ready to drive (page object).

Example (Full Working Code — Corrected)

javapackage org.sujalkhot.day14;

import java.awt.Dimension;
import java.awt.Toolkit;
import java.util.Arrays;

import com.microsoft.playwright.Browser;
import com.microsoft.playwright.BrowserContext;
import com.microsoft.playwright.BrowserType;
import com.microsoft.playwright.Page;
import com.microsoft.playwright.Playwright;

public class BrowserFactory {

    // Encapsulation: private static — managed only inside this class
    private static Playwright playwright;
    private static Browser browser;

    public static Page initBrowser(String browserName, boolean headless, boolean fullScreen) {

        // Step 1: Start Playwright server
        playwright = Playwright.create();

        // Step 2: Get physical screen size of host machine (for Firefox/WebKit fullscreen)
        Dimension screenSize = Toolkit.getDefaultToolkit().getScreenSize();
        int width = (int) screenSize.getWidth();
        int height = (int) screenSize.getHeight();

        // Step 3: Set base launch options
        BrowserType.LaunchOptions option = new BrowserType.LaunchOptions().setHeadless(headless);

        // Step 4: For Chromium-based browsers, add --start-maximized if fullScreen is true
        if (fullScreen && (browserName.equalsIgnoreCase("Chromium")
                || browserName.equalsIgnoreCase("Chrome")
                || browserName.equalsIgnoreCase("edge"))) {
            option.setArgs(Arrays.asList("--start-maximized"));
        }

        // Step 5: Launch the correct browser
        switch (browserName.toLowerCase()) {

            case "chromium":
                browser = playwright.chromium().launch(option);
                break;

            case "chrome":
                option.setChannel("chrome");
                browser = playwright.chromium().launch(option);
                break;

            case "edge":
                option.setChannel("edge");
                browser = playwright.chromium().launch(option);
                break;

            case "firefox":
                browser = playwright.firefox().launch(option);
                break;

            case "webkit":
                browser = playwright.webkit().launch(option);
                break;

            default:
                throw new IllegalArgumentException("Invalid Browser: " + browserName);
        }

        // Step 6: Create Browser Context
        BrowserContext context;

        if (fullScreen && (browserName.equalsIgnoreCase("chromium")
                || browserName.equalsIgnoreCase("chrome")
                || browserName.equalsIgnoreCase("edge"))) {
            // null viewport lets --start-maximized take effect
            context = browser.newContext(new Browser.NewContextOptions().setViewportSize(null));

        } else if (fullScreen && (browserName.equalsIgnoreCase("firefox")
                || browserName.equalsIgnoreCase("webkit"))) {
            // Firefox/WebKit: manually set viewport to screen size
            context = browser.newContext(new Browser.NewContextOptions().setViewportSize(width, height));

        } else {
            context = browser.newContext(new Browser.NewContextOptions());
        }

        // Step 7: Open a new tab and return it
        return context.newPage();
    }

    public static void closeBrowser() {
        if (playwright != null) {
            playwright.close(); // Closes all browsers and releases memory
        }
    }
}

Interview Questions

Q1: What is the Browser Factory design pattern in Playwright?

Answer: It is a design pattern where browser creation logic is centralized in one class. It avoids code duplication, supports multiple browsers, and makes the framework easy to maintain.

Q2: Why do Firefox and WebKit not support --start-maximized?

Answer: --start-maximized is a Chromium-specific browser argument. Firefox and WebKit have their own rendering engines and do not accept Chromium flags. To simulate fullscreen, you set the viewport size equal to the screen dimensions.

Q3: What is the role of BrowserContext in Playwright?

Answer: BrowserContext is like an incognito session. It is isolated from other contexts — separate cookies, storage, and sessions. Each context can have multiple pages (tabs).

Q4: Why are playwright and browser declared as private static?

Answer: private ensures encapsulation — these variables cannot be accessed or modified from outside the class. static means they are shared at the class level and don't need an object to be used.

Q5: What happens if you don't call playwright.close()?

Answer: Browser processes keep running in the background, wasting system memory and CPU. Always close Playwright after tests finish.

BrowserFactory is a reusable class that launches any browser with one method call, handles fullscreen differences, and follows OOP principles.





Lecture 15 — Locators in Playwright


What is a Locator?


A Locator is like the address of a web element on a page — it tells Playwright exactly which element to find and interact with.
Think of a web page like a city and elements like houses. A Locator is the GPS address that leads you to the right house.
Playwright Locators are smart — they automatically:

Wait for the element to appear (auto-wait).
Retry if the element is not found immediately.
Re-find the element before every action (in case the page reloaded or DOM changed).
Ensure the element is visible before clicking/typing.






Why Do We Use Locators?


Web pages are dynamic — elements load at different times.
Without Locators, your test would fail if the element takes a second to appear.
Locators handle timing, retrying, and DOM changes automatically — less flaky tests.

Interview Questions

Q1: What is a Locator in Playwright?

Answer: A Locator is an object that identifies a web element and performs actions on it. It auto-waits, retries, and re-finds elements before every action, making tests more stable.

Q2: What is the difference between absolute and relative XPath?

Answer: Absolute XPath starts from the root HTML element and traverses the full DOM tree — it breaks easily if any parent element changes. Relative XPath starts directly from the target element using // and is much more stable.

Q3: Why is getByRole() the most preferred locator?

Answer: It is based on ARIA accessibility roles, making tests semantic and aligned with how users and screen readers interact with the page. It is also the most resilient to UI changes.

Q4: What is data-testid and why is it used?

Answer: data-testid is a custom HTML attribute added by developers specifically for automation. It doesn't change with UI redesigns, making it the most stable locator for enterprise frameworks.

Q5: What is the difference between XPath index and Playwright's .nth() index?

Answer: XPath index starts at 1 — e.g., (//input)[1] is the first element. Playwright's .nth() starts at 0 — e.g., .nth(0) is the first element. This is a common interview trick question.

Q6: What is the difference between page.locator("text=Login") and page.getByText("Login")?

Answer: Both find elements by visible text. getByText() is the newer, recommended built-in method. locator("text=Login") is an older CSS-text syntax. Both work, but getByText() is preferred.

Q7: Can CSS selectors traverse to parent elements?

Answer: No. CSS selectors can only go downward (child/descendant). To access parent or sibling elements, you must use XPath.

Role Locator — Deep Dive


What is a Role Locator?


A Role Locator finds elements using their ARIA role (Accessibility Role) and their visible name (the text you see on screen).
It locates elements exactly the way screen readers and real users see them — not by technical IDs or classes.
ARIA stands for Accessible Rich Internet Applications — a standard that makes web pages readable for people with disabilities (e.g., screen readers read out button names aloud).


Memory Trick: Think of ARIA like the voice description of a button. A screen reader says "Login Button". Role Locator uses that same description to find it.


Why Do We Use Role Locators?


⚡ Most stable — If the developer changes the CSS class or ID of a button but keeps the visible text "Login", your test still passes.
Aligns with real user behavior — finds elements the way humans see them.
Recommended first by the Playwright team.
Works even when elements use div or span tags with role="button" instead of a real <button> tag.



Key Points to Remember


⚡ Always use setName() to be specific — without it, Playwright finds all elements with that role, and if there are multiple it throws an error.
⚡ If there are 2 or more elements with the same role + name (e.g., two "Password" fields), Playwright throws a strict mode error and shows the error in the console — it also gives suggestions on how to fix it.
⚡ Playwright tells you how many elements matched and where they are in the error message — read the console carefully.
Playwright finds the role using: aria-label, aria-labelledby, label text, inner text, placeholder, or associated <label> tag.
🚫 Do NOT use .nth() in real frameworks as a fix for duplicate elements — the index can change if the page layout changes.

Example — Full Login Automation Using Role Locators

javapackage org.sujalkhot.day14;

import com.microsoft.playwright.Page;
import com.microsoft.playwright.options.AriaRole;

public class LaunchBrowseTest {

    public static void main(String[] args) {

        // Launch Edge browser (non-headless, no fullscreen)
        Page page = BrowserFactory.initBrowser("edge", false, false);

        // Navigate to the test website
        page.navigate("https://swautomationtester-dot.github.io/IshaDemoV1/");

        // Find the "Username" textbox by role + name and fill it
        page.getByRole(AriaRole.TEXTBOX, new Page.GetByRoleOptions().setName("Username")).fill("admin");

        // Find the "Password" textbox by role + name and fill it
        page.getByRole(AriaRole.TEXTBOX, new Page.GetByRoleOptions().setName("Password")).fill("admin123");

        // Find the "Login" button by role + name and click it
        page.getByRole(AriaRole.BUTTON, new Page.GetByRoleOptions().setName("Login")).click();

        // Print the current page title
        System.out.println(page.title());

        // Wait 2 seconds before closing (for visual verification)
        page.waitForTimeout(2000);

        // Close the browser
        BrowserFactory.closeBrowser();
    }
}

Interview Questions

Q1: What is ARIA and why is it relevant to Playwright's Role Locator?

Answer: ARIA (Accessible Rich Internet Applications) is a standard that adds accessibility metadata to HTML elements. Playwright's getByRole() uses ARIA roles to find elements — the same way screen readers and assistive technologies do. This makes tests more human-readable and resilient.

Q2: What happens when multiple elements match the same role + name in Playwright?

Answer: Playwright throws a strict mode violation error. It tells you how many elements matched, where they are, and suggests fixes. Solutions include using data-testid, adding a more specific name, using an ancestor XPath, or (as a last resort) .nth().

Q3: What is the difference between textContent() and innerText()?

Answer: textContent() returns all text inside an element including hidden text and extra spaces. innerText() returns only the visible text (what the user actually sees) and handles whitespace better. In real projects, always .trim() after textContent().

Q4: Why should you avoid using .nth() in real frameworks?

Answer: .nth() selects an element by its position in the DOM. If the UI changes (e.g., a new product is added before the target), the index shifts and the test breaks. It is not stable for long-term automation.

Q5: How does Playwright determine the visible name of an element for getByRole()?

Answer: Playwright checks in this order: aria-label attribute, aria-labelledby reference, associated <label> tag, inner text, and placeholder attribute.

Q6: What is the difference between page.title() and page.url()?

Answer: page.title() returns the browser tab title (the <title> tag value). page.url() returns the current full URL of the page.

Q7: What does locator.count() return if no elements are found?

Answer: It returns 0 — it does not throw an error. This makes it safe to use for checking if elements exist before interacting with them.

Q8: When would you use locator.getAttribute("href") in a real project?

Answer: To verify that a link points to the correct URL. For example: assert locator.getAttribute("href").equals("/home").

1. XPath Best Practices — Parent-Child vs .. Traversal


What Is the Issue?

When writing XPath, sometimes you need to go up to a parent element and then come back down to find a sibling element. There are two ways to do this:

ApproachExampleRecommended?Using .. (parent shortcut)//input[@id='email']/..❌ AvoidUsing Parent-Child path//div[@class='form-group']/input✅ Prefer


Why Avoid .. Traversal?


.. means "go to parent" — it works but is harder to read and more fragile.
Long chains of ../.. (like ../..//button) become difficult to maintain.
Other team members struggle to understand what element is being targeted.


xpath❌ Avoid:
//div[text()='Isha Bolt T-Shirt']/../..//button[text()='Add to cart']

✅ Prefer:
//div[@class='product-card'][.//div[text()='Isha Bolt T-Shirt']]//button[text()='Add to cart']


Memory Trick


.. = going up the stairs blindly.
Parent-Child = drawing a clear map from the front door.



Always write XPath that someone else can read and understand without running it.

Playwright can wait at different points during page loading. You choose based on how much of the page your test needs.

Interview Questions

Q1: What is the difference between DOMCONTENTLOADED, LOAD, and NETWORKIDLE?

Answer: DOMCONTENTLOADED waits only for HTML to be parsed — no images or API calls. LOAD waits for HTML + images + CSS + fonts. NETWORKIDLE waits until all network requests stop, including API calls. Use LOAD for most cases, NETWORKIDLE for dynamic/API-driven pages.

Q2: What is Auto-Waiting in Playwright?

Answer: Playwright automatically checks that an element exists, is visible, is enabled, and is stable before performing any action like click() or fill(). This reduces the need for explicit waits and makes tests more reliable.

Q3: Why should we avoid Thread.sleep() in Playwright?

Answer: Thread.sleep() always waits for the full duration even if the element loads earlier, making tests unnecessarily slow. It is also unreliable — if the element takes longer than the sleep duration, the test still fails. Playwright's auto-waiting handles this automatically.

Q4: What is the difference between fill() and type() in Playwright?

Answer: fill() clears existing text and enters a new value in one operation. type() appends text character by character without clearing existing content. Use fill() for most scenarios and type() only when simulating real keyboard input is required.

Q5: How do you handle a loading spinner in Playwright?

javapage.waitForSelector(".spinner",

Q6: Why is Locator preferred over ElementHandle for dynamic elements?

Answer: A Locator re-queries the DOM every time it performs an action, so it handles dynamic updates correctly. An ElementHandle stores a fixed reference to an element — if the DOM updates (e.g., React re-renders), the handle becomes stale and throws an error.

Q7: What is waitForNavigation() and when do you use it?

Answer: waitForNavigation() wraps an action that triggers a page navigation and waits for the navigation to complete before continuing. Use it when clicking a link or button that opens a new page.

Q8: What is the default timeout in Playwright and how do you change it?

Answer: Default timeout is 30 seconds. You can change it globally with page.setDefaultTimeout(ms) for all actions, or page.setDefaultNavigationTimeout(ms) for navigation-only.

Q9: How do you select and verify a radio button in Playwright?

Answer: Select using page.check("input[value='male']") and verify using page.isChecked("input[value='male']") which returns true or false.

Q10: What is the Retry Mechanism in Playwright?

Answer: Playwright automatically retries actions (like click(), fill()) until the element becomes ready or the timeout is reached. This handles dynamic pages where elements load after a delay without needing manual retry logic.

2. Handling Multiple Elements


Selenium Way — findElements()

javaList<WebElement> products = driver.findElements(By.className("product"));

for (WebElement p : products) {
    System.out.println(p.getText());
}


Returns a List<WebElement>.
Returns empty list if nothing found — no exception thrown.



Playwright Way — locator().all()

javaList<Locator> products = page.locator(".product").all();

for (Locator p : products) {
    System.out.println(p.textContent());
}


Returns a List<Locator>.
Each item in the list is a Locator for one matching element.
Use when you need to perform actions on each element.



Faster Way — allTextContents()

When you only need the text of all matching elements (no actions needed):

javaList<String> productNames = page.locator(".prod-title").allTextContents();
System.out.println(productNames);
// Output: [Laptop, Mobile, Headphones]


Returns a List<String> directly.
No loop needed.
Cleaner and faster code.

3. Common Locator Mistakes


❌ Space vs . in CSS Class Selectors

java// ❌ WRONG — Space between classes = "btn primary" is treated as DESCENDANT selector
//    Looks for an element with class "primary" INSIDE an element with class "btn"
page.locator(".btn primary").click();  // Throws error: element not found

// ✅ CORRECT — No space = element must have BOTH classes "btn" AND "primary"
page.locator(".btn.primary").click();


⚡ Interview Tip: .btn.primary = one element with two classes. .btn primary = an element with class primary that is a child of .btn. These are completely different!




Handling Duplicate Elements (Same Locator Matches Multiple)

java// ❌ Throws error if more than one submit button exists
page.locator("[type='submit']").click();

// ✅ Use first(), last(), or nth() to be specific
page.locator("[type='submit']").first();     // First match
page.locator("[type='submit']").nth(1);      // Second match (0-indexed)
page.locator("[type='submit']").nth(2);      // Third match
page.locator("[type='submit']").last();      // Last match


⚡ Note: .nth() is 0-indexed in Playwright — .nth(0) is the first element, .nth(1) is the second.

5. page.onDialog() vs page.onceDialog()


page.onDialog() — Permanent Listener

javapage.onDialog(dialog -> {

});

Risk: may accidentally accept dialogs you didn't expect.

javapage.onceDialog(dialog -> {
    System.out.println("Alert: " + dialog.message());

Handles only the next dialog.
Automatically removed after use.
Safer and more predictable.

⚡ CRITICAL: Always Register the Listener BEFORE Clicking

java// ✅ CORRECT ORDER — Register first, then click
page.onceDialog(dialog -> {
    dialog.accept();
});
page.click("#alertButton");   // Alert appears → listener is ready → handles it

// ❌ WRONG ORDER — Click first, then try to register
page.click("#alertButton");   // Alert appears IMMEDIATELY — no listener yet!
page.onceDialog(dialog -> {   // Too late — browser is blocked, test hangs
    dialog.accept();
});

Why?


JavaScript alerts are blocking — when an alert appears, the browser freezes and no further actions can be executed.
Playwright must already be listening before the alert appears.
Wrong order = test hangs → timeout → test fails.


Memory Trick:

First Listen  →  onceDialog()
Then Click    →  click()
Alert Appears →  Playwright handles it automatically

7. Modern Modal / Toast Notifications


What Are These?


Not browser alerts — they are HTML elements built into the web page.
They can be inspected in DevTools like any other element.
Handle them using regular Locators — not onDialog().


java// Click a modal button

How to Find a Toast Message Element (It Disappears Quickly)

Since toast messages are visible for only a short time, use this trick to freeze the page and inspect the element:


Open DevTools (F12)
Go to Sources tab
Expand Event Listener Breakpoints
Enable Keyboard (or Mouse events)
Perform the action that triggers the toast
When the toast appears, press F8 to pause/freeze the page
Switch to the Elements tab and inspect the toast element
Find its class or ID (e.g., #toastContainer, .toast.success.show)
Press the Resume (▶) button to unfreeze



⚡ When OK, Cancel, and Error toasts share the same ID, use a more specific XPath:
page.locator("//*[@id='toastContainer']/div").innerText()

Method 2: Using BrowserContext Credentials (Recommended ✅)

javapackage org.sujalkhot.day19;

import java.util.Arrays;
import com.microsoft.playwright.*;

public class HandlingAuth {

    public static void main(String[] args) {

        Playwright playwright = Playwright.create();

        Browser browser = playwright.chromium().launch(
            new BrowserType.LaunchOptions()
                .setHeadless(false)
                .setArgs(Arrays.asList("--start-maximized"))
                .setSlowMo(1000)
        );

        // ✅ Recommended: Pass credentials at context level
        BrowserContext context = browser.newContext(
            new Browser.NewContextOptions()
                .setViewportSize(null)
                .setHttpCredentials("admin", "admin")  // Username, Password
        );

        Page page = context.newPage();
        page.navigate("https://the-internet.herokuapp.com/basic_auth");

        // The auth popup is bypassed — page loads directly
    }
}

Interview Questions

Q1: How is page.locator() different from Selenium's findElement()?

Answer: Selenium's findElement() immediately searches for the element and throws an exception if not found. Playwright's locator() creates a deferred reference — it searches for the element only when an action is performed, with auto-waiting built in.

Q2: How do you get all elements matching a locator in Playwright?

Answer: Use page.locator(".selector").all() which returns a List<Locator>. If you only need the text values, use allTextContents() which returns a List<String> without needing a loop.

Q3: What is the difference between .btn primary and .btn.primary in CSS?

Answer: .btn.primary selects an element that has both classes "btn" AND "primary". .btn primary (with a space) is a descendant selector — it selects an element with class "primary" that is nested inside an element with class "btn". These are completely different selectors.

Q4: What is the difference between page.onDialog() and page.onceDialog()?

Answer: page.onDialog() is a permanent listener that handles all dialogs throughout the page lifecycle. page.onceDialog() handles only the next dialog and is automatically removed after use, making it safer for specific test scenarios.

Q5: Why must you register the dialog listener BEFORE clicking the button that triggers an alert?

Answer: JavaScript alerts are blocking — when they appear, the browser freezes and no further JavaScript can execute. Playwright must already be listening for the dialog event before the alert appears, otherwise the test hangs and times out.

Q6: How do you handle a Prompt alert in Playwright?

Answer: Use dialog.accept("your text") inside the dialog listener. The text passed to accept() is typed into the prompt input box and OK is clicked. dialog.dismiss() clicks Cancel without entering text.

Q7: What is the difference between a browser alert and a modern modal/toast?

Answer: Browser alerts (Simple, Confirmation, Prompt) are native browser popups — they are not in the DOM and cannot be inspected. Modern modals and toasts are HTML elements built into the page — they can be inspected and handled using regular Playwright locators.

Q8: How does Playwright handle HTTP Basic Authentication?

Answer: HTTP Auth popups are browser-native and not part of the DOM. Playwright bypasses them by providing credentials at the BrowserContext level using .setHttpCredentials("username", "password") when creating the context, before navigating to the page.

Q9: Why is passing credentials in the URL (like https://user:pass@site.com) not recommended?

Answer: Credentials in the URL are exposed in browser history, server logs, and test reports — this is a security risk. The recommended approach is BrowserContext.setHttpCredentials().

Q10: What does allTextContents() return and when should you use it?

Answer: It returns a List<String> containing the text content of all matching elements. Use it when you only need to read text values (no actions needed) — it's faster and cleaner than looping through locator().all().
