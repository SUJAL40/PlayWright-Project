#) Selenium Architecture:

HTTP Connection 

				Request
		-------------------------------> 
				Response
Client 	<-------------------------------	Server
			Connection Terminate

#) Key Points:

- Selenium sends each command as a separate HTTP request and receives a JSON response.
- So, every action, such as opening a browser, clicking an element, or sending keys in a text box,
  is sent as a separate HTTP request.
- After completion of every request, the connection between server & client will be terminated,
  which needs to be re-established for the next request.
- Connection termination after every request results in slower execution, which introduces a layer of flakiness

#) Playwright Architecture

    Websocket connection

	Client1 ----Request---->  Cloud
	Client1 <---Handshake---  Cloud
	Client1 <---Web socket--> Cloud

- Playwright on other hand communicate all requests through a single web socket connection,
  which stay in place until test execution is completed.
- This reduces the points of failure and allows commands to be sent quickly on a single connection.
- A primary reason for Playwright's speed especially when compared to selenium is its architecture
- Playwright makes use of WebSockets instead of HTTP connection to comm. with end browser send & receive request.

#) Playwright Architecture

- Establishing comm. with websocket is extremely beneficial as it create one connection & 
  keep the connection alive until either disconnects explicitly.
- In this case, that connection will soon when all the test cases are executed.
- The complete process can be documented as below:

   a) A websocket conn. request is sent from client to server
   b) If the configurational parameter are correct, conn. is accepted
   c) The server acknowledges the conn. establishment to client
   d) The test execution begins on same conn.
   e) All the test cases use same connection for their execution.
   f) Once the execution is complete, the connection is disbanded.

											Exception Handling in Java
											
#) What is Exception Handling?
-> An exception is an unwanted event that disrupts the normal flow of a program.
	Exception handling is a mechanism to handle runtime errors so that the program can continue execution.

#) Why Do We Need Exception Handling?

Scenario:
Suppose a Java program contains 10 statements and an exception occurs at statement 5.

Without exception handling:
Statements 6 to 10 will not be executed.
Program execution terminates abnormally.

With exception handling:
The exception is handled.
The remaining statements continue to execute normally.

Therefore, exception handling helps maintain the normal flow of the program.

#) Types of Exceptions in Java

	Java provides built-in exceptions and also allows users to create custom exceptions.

	1. Built-in Exceptions
	Built-in exceptions are predefined exceptions available in Java libraries.
	Examples:
	
	ArithmeticException : Thrown when an arithmetic operation causes an error.
	Example:
	int result = 10 / 0;

	ArrayIndexOutOfBoundsException : Thrown when accessing an array with an invalid index.
	Example:
	int[] arr = {1, 2, 3};
	System.out.println(arr[5]);

	ClassNotFoundException : Thrown when the JVM cannot find a class definition.

	FileNotFoundException : Thrown when a file is not found or cannot be opened.

	IOException : Thrown when an input/output operation fails.

	InterruptedException : Thrown when a thread is interrupted while waiting, sleeping, or processing.

	NoSuchFieldException : Thrown when a specified field does not exist in a class.

	NoSuchMethodException : Thrown when a specified method does not exist.

	NullPointerException : Thrown when accessing members of a null object reference.
	Example:
	String s = null;
	System.out.println(s.length());

	NumberFormatException : Thrown when converting a string to a numeric type fails.
	Example:
	Integer.parseInt("abc");

	RuntimeException : Represents exceptions that occur during program execution.

	StringIndexOutOfBoundsException : Thrown when a string index is negative or exceeds string length.
	Example:
	String str = "Java";
	System.out.println(str.charAt(10));

#) Exception Handling Keywords :

- try
	Used to define a block of code that may throw an exception.
	Cannot be used alone.
	Must be followed by catch or finally.

Syntax:
	try {
	// risky code
	}

catch
	Used to handle exceptions.
	Must be preceded by a try block.
	Can be followed by a finally block.

Syntax:
	catch(Exception e) {
	// handling code
	}

finally
	Used to execute important cleanup code.
	Executes whether an exception occurs or not.

Syntax:
	finally {
	// cleanup code
	}

throw
	Used to manually throw an exception.

Syntax:
	throw new ArithmeticException("Divide by zero not allowed");

throws
	Used to declare exceptions in a method signature.
	Informs the caller that a method may throw an exception.

Syntax:
	void readFile() throws IOException {
	}

#) Keywords Summary

package org.sujalkhot;

import com.microsoft.playwright.Browser;
import com.microsoft.playwright.BrowserType;
import com.microsoft.playwright.Page;
import com.microsoft.playwright.Playwright;

public class LaunchBrowser {

	public static void main(String[] args) {

		// Create Playwright instance
		// Playwright is the entry point for launching and controlling browsers
		Playwright pw = Playwright.create();

		// Launch Chromium browser
		// By default Playwright runs in headless mode (without UI)
		// setHeadless(false) opens the browser with UI (headed mode)
		Browser browser = pw.chromium()
				.launch(new BrowserType.LaunchOptions().setHeadless(false));

		// Create a new browser tab (Page)
		Page page = browser.newPage();

		// Navigate to the application URL
		page.navigate("https://www.saucedemo.com");

		// Print the title of the current webpage in the console
		System.out.println(page.title());

		// Enter username in the username textbox
		// '#' represents ID selector in CSS
		page.fill("#user-name", "standard_user");

		// Enter password in the password textbox
		page.fill("#password", "secret_sauce");

		// Click on Login button
		page.click("#login-button");

		// Close the browser window
		browser.close();

		// Close Playwright instance and release resources
		pw.close();
	}
}


# Playwright Codegen (Recorder)

- Playwright provides a Codegen (Recorder) tool that records user actions and 
  automatically generates Playwright automation code.

## Steps to Use Codegen

1. Open Terminal / Command Prompt.
2. Navigate to your Playwright project directory.

Example:

cd "C:\Users\sujal\OneDrive\Documents\PlayWright Project\PlayWright-Project\playwright"

3. Ensure the following prerequisites are available:

- pom.xml
- src folder
- target folder
- Playwright dependencies installed

4. Run the following command:

mvn exec:java -e -Dexec.mainClass=com.microsoft.playwright.CLI -Dexec.args="codegen https://swautomationtester-dot.github.io/IshaDemoSite/index.html"

## What Does This Command Do?

- Launches Playwright Codegen (Recorder).
- Opens a browser and navigates to the specified URL.
- Records user actions such as click, fill, select, and navigation.
- Automatically generates Playwright Java code.
- Helps create automation scripts quickly without writing locators manually.

## Benefits

- Generates reliable locators.
- Speeds up script creation.
- Useful for beginners.
- Helps inspect elements and create assertions.

- Instead of above cmd we need to add
	page.navigate("https://swautomationtester-dot.github.io/IshaDemoSite/index.html");
	page.pause()

What is Multiple Browser Context?
	Playwright allows multiple Browser Contexts inside a single browser instance.
	Each Browser Context acts like a separate Incognito Window.

Why Use Multiple Browser Contexts?
	Test multiple users simultaneously.
	Test chat applications.
	Test Admin and User interactions.
	Save memory by using one browser with multiple sessions.


| Browser                            | Browser Context                  |
| ---------------------------------- | -------------------------------- |
| Actual browser instance            | Isolated browser session         |
| More memory usage                  | Lightweight                      |
| Separate browser launch required   | Created from same browser        |
| Used for different browser engines | Used for multiple users/sessions |

- Browser Factory: 

	Browser Factory is a design pattern used to create and manage browser instances from a single place.
	It helps launch different browsers (Chromium, Firefox, WebKit) based on configuration.
	Avoids writing browser launch code multiple times.

- firefox & Webkit do not support --start-maximized
- They rely on viewport size
- This ensure true fullscreen on all broswer so, we need to set dimensions
- BrowserFactory class, encapsulation is achieved through data hiding using access modifiers. 
	eg. private static Playwright playwright;

- Steps :
	step 1: Create Playwright instance
	step 2: Launch Browser (Chromium,Firefox,Webkit)
	step 3: Create Context
	step 4: Open a Page (Tab)
	step 5: Navigate to URL

eg.
package org.sujalkhot.day14;

import java.awt.Dimension;
import java.awt.Toolkit;
import java.util.Arrays;

import com.microsoft.playwright.Browser;
import com.microsoft.playwright.BrowserContext;
import com.microsoft.playwright.BrowserType;
import com.microsoft.playwright.Page;
import com.microsoft.playwright.Playwright;

public class BrowserFactory {

	// Encapsulation: Storing instances at the class level so they can be managed globally.
	private static Playwright playwright;
	private static Browser browser;

	// Method to initialize and return a new browser page based on provided configurations.
	public static Page initBroswer(String browserName, boolean headless, boolean fullScreen) {

		// 1. Start the Playwright server
		playwright = Playwright.create();

		// 2. Get the physical screen dimensions of the host machine (used later for Firefox/Webkit)
		Dimension screenSize = Toolkit.getDefaultToolkit().getScreenSize();
		int width = (int) screenSize.getWidth();
		int height = (int) screenSize.getHeight();

		// Note: This line launches an orphaned Chromium instance that is never closed or assigned. 
		// You likely want to remove this line as it wastes memory.
		playwright.chromium().launch(new BrowserType.LaunchOptions().setHeadless(false).setSlowMo(1000));

		// 3. Set the base launch options (primarily the headless mode flag)
		BrowserType.LaunchOptions option = new BrowserType.LaunchOptions().setHeadless(headless);

		// 4. If fullScreen is requested and the browser is Chromium-based, pass the maximize argument
		// for chromium we have to set the args
		if (fullScreen && (browserName.equalsIgnoreCase("Chromium") || browserName.equalsIgnoreCase("Chrome")
				|| browserName.equalsIgnoreCase("edge"))) {
			option.setArgs(Arrays.asList("--start-maximized"));
		}

		// 5. Launch the specific browser engine requested by the user
		switch (browserName.toLowerCase()) {

		case "chromium":
			// Launches default open-source Chromium
			browser = playwright.chromium().launch(option);
			break;

		case "chrome":
			// Launches branded Google Chrome installed on the machine
			option.setChannel("chrome");
			browser = playwright.chromium().launch(option);
			break;

		case "edge":
			// Launches branded Microsoft Edge installed on the machine
			option.setChannel("edge");
			browser = playwright.chromium().launch(option);
			break;

		case "firefox":
			// Launches Mozilla Firefox
			browser = playwright.firefox().launch(option);
			break;

		case "webkit":
			// Launches WebKit (the engine behind Safari)
			browser = playwright.webkit().launch(option);
			break;

		default:
			// Fails safely if an unsupported browser name is passed
			throw new IllegalArgumentException("Invalid Browser : " + browserName);

		}

		// 6. Configure the isolated Browser Context (like an incognito window)
		BrowserContext context;

		if (fullScreen && (browserName.equalsIgnoreCase("Chromium") || browserName.equalsIgnoreCase("Chrome")
				|| browserName.equalsIgnoreCase("edge"))) {
			// For Chromium browsers, viewport size must be null so the "--start-maximized" arg takes over
			context = browser.newContext(new Browser.NewContextOptions().setViewportSize(null));
			
		} else if (fullScreen && (browserName.equalsIgnoreCase("firefox") || browserName.equalsIgnoreCase("webkit"))) {
			// Firefox/Webkit don't support "--start-maximized", so we force the viewport to match screen dimensions
			context = browser.newContext(new Browser.NewContextOptions().setViewportSize((int) width, (int) height));
			
		} else {
			// Default context if fullScreen is false
			context = browser.newContext(new Browser.NewContextOptions());
		}

		// 7. Open a new tab/page in the configured context and return it
		return context.newPage();
	}
	
	public static void closeBrowser() {
    // Check for null to prevent a NullPointerException
    if (playwright != null) {
        // Closes the Playwright connection and terminates all active browsers
        playwright.close();
    }
}
}

Playwright Page Load States

1. DOM Content Loaded
	- Waits until HTML page is loaded and parsed.
	- Does not wait for images, CSS, or API calls.
	- Faster than other load states.
	- Use when page structure is enough for your test.

Example:
	page.waitForLoadState(LoadState.DOMCONTENTLOADED);

2. Load
	- Waits until the page and its resources are loaded.
	- Includes images, CSS, fonts, etc.
	- Most commonly used page load state.

Example:
	page.waitForLoadState(LoadState.LOAD);

3. Network Idle
	- Waits until network activity stops.
	- Waits for API calls and background requests to finish.
	- Can make tests slower.
	- Use only when data is loaded from APIs.

Example:
	page.waitForLoadState(LoadState.NETWORKIDLE);

Easy Memory Trick

	DOM Content Loaded  -> HTML loaded
	Load                -> HTML + Images + CSS loaded
	Network Idle        -> API calls finished

Dynamic Elements

What are Dynamic Elements?
- Dynamic elements are elements that do not appear immediately on the page.
- They may change or load after some time.

Examples:
1. Load after API calls
2. Appear after animations
3. Become visible or enabled later
4. Change text or attributes
5. Get re-rendered by frameworks like React

Example:
- Loader spinner disappears after data is loaded.
- Submit button becomes enabled after entering valid data.
- Search results appear after typing in the search box.

How Playwright Handles Waits

Playwright automatically waits before performing actions.

When you write:

page.click("button");

Playwright automatically waits for:

- Element exists in the page
- Element is visible
- Element is enabled
- Element is stable (not moving or animating)

This feature is called Auto-Waiting.

Benefits:
- Less flaky tests
- No need for most manual waits
- Faster and more reliable automation

--------------------------------------------------

- Can still fail if element takes longer to load

- Use Playwright Auto-Waiting
- Use waitFor() when required

Retry Mechanism in Playwright

What is Retry Mechanism?

- Playwright automatically retries actions until they succeed or the timeout is reached.
- This makes tests more stable for dynamic web applications.

Playwright Retries When:

- Element is not ready yet
- Element is not visible
- Element is not enabled
- Page is re-rendering (React, Angular, Vue)
- DOM updates dynamically

--------------------------------------------------

Default Timeout

- Default timeout is 30 seconds.
- Playwright keeps retrying during this time.

Example:

page.click("#submit");

Playwright retries automatically until:
- Element becomes available
- Or timeout is reached

--------------------------------------------------

Custom Timeout

Use a custom timeout for a specific action.

Example:

page.click(
    "#submit",
    new Page.ClickOptions().setTimeout(5000)
);

- Retries for 5 seconds before failing.

File Upload in Playwright (Java)

What is File Upload?

- File upload is used to attach files such as PDF, images, documents, etc.
- Playwright directly interacts with the file input element.
- No need to handle OS file picker popups manually.

HTML Example:

	<input type="file" id="uploadFile">

--------------------------------------------------

setInputFiles() Method

- Used to upload files.
- Recommended and easiest way in Playwright.

--------------------------------------------------

Upload Single File

Example:

	page.setInputFiles("#uploadFile",
		Paths.get("C:/files/resume.pdf"));

OR

	page.locator("#uploadFile")
		.setInputFiles(Paths.get("C:/files/resume.pdf"));

What happens?
- Playwright selects the file automatically.
- The file is attached to the input element.

--------------------------------------------------

Upload Multiple Files

Example:

	page.setInputFiles("#uploadFile",
		new Path[] {
			Paths.get("C:/files/a.pdf"),
			Paths.get("C:/files/b.pdf")
		});

What happens?
- Multiple files are uploaded at the same time.
- Works only if the HTML input supports multiple files.

--------------------------------------------------

Clear Uploaded File

Example:

	page.setInputFiles("#uploadFile", new Path[0]);

What happens?
- Removes the currently selected/uploaded file(s).
- Resets the file input field.

--------------------------------------------------

Best Practice

✓ Use setInputFiles() for file uploads.
✓ No need for Robot class or AutoIT.
✓ No need to handle system file dialogs.
✓ Direct and reliable approach in Playwright.

--------------------------------------------------

Interview Answer

In Playwright, file uploads are handled using setInputFiles(). 
It allows uploading single or multiple files directly to a file input
 element without interacting with the operating system's file picker dialog.

#) File Upload using FileChooser in Playwright (Java)

	When is FileChooser Used?

	- Used when clicking an Upload button opens the system file picker.
	- The page does not expose a direct <input type="file"> element.
	- Playwright handles the file upload without interacting with the OS dialog.

--------------------------------------------------

Why Use FileChooser?

In Selenium:
- Robot Class
- AutoIT
- OS dialog handling

In Playwright:
- No need to handle OS file dialogs.
- Use FileChooser event and set files directly.

--------------------------------------------------

Example Scenario

HTML:

<button id="uploadBtn">Upload</button>

When the button is clicked:
- System file picker opens.

--------------------------------------------------

Handle FileChooser

Example:

	FileChooser fileChooser = page.waitForFileChooser(() -> {
		page.click("#uploadBtn");
	});

	fileChooser.setFiles(
		Paths.get("C:/files/resume.pdf")
	);

What Happens?

1. Playwright waits for the file chooser event.
2. Clicks the Upload button.
3. Captures the file chooser.
4. Uploads the file programmatically.
5. No interaction with OS popup is required.

--------------------------------------------------

Best Practice

✓ Use setInputFiles() when <input type="file"> is available.
✓ Use FileChooser when clicking a button opens a file picker.
✓ Avoid Robot Class and AutoIT in Playwright.

--------------------------------------------------

Interview Answer

FileChooser is used when clicking an Upload button opens a system file picker. 
Playwright captures the file chooser event using waitForFileChooser() and uploads files 
using setFiles() without interacting with the operating system dialog.

File Download in Playwright (Java)

What is File Download?

- Used when clicking a button or link downloads a file.
- Playwright provides built-in support for handling downloads.
- No browser profile configuration is required.

--------------------------------------------------

Download Triggered by Click

When a click starts a download, use:

page.waitForDownload()

Example:

Download download = page.waitForDownload(() -> {
    page.click("#downloadBtn");
});

What Happens?

1. Playwright waits for a download event.
2. Clicks the download button.
3. Captures the downloaded file.

--------------------------------------------------

Useful Download Methods

- User Website : https://the-internet.herokuapp.com/download

- Returns the temporary file location.
- Mainly used for verification.

- Returns the original filename sent by the server.

Example:

String fileName = download.suggestedFilename();

3. download.saveAs()

- Saves the downloaded file to a custom location.

Example:

download.saveAs(

- File is saved in the specified folder.
- Original filename is preserved.

✓ Use download.saveAs() to store files in a custom location.
✓ Use suggestedFilename() to get the original filename.
✓ No browser profile configuration is needed.

Playwright provides built-in support for file upload and download. 
We use setInputFiles() for uploads, waitForFileChooser() for upload popups, and waitForDownload() with saveAs() for file downloads.

* page.waitForDownload() is a download listener in Playwright.
* It waits for a file download to start after an action such as clicking a Download button.

Example:
	page.waitForDownload(() -> {
	downloadButton.click();
	});

* In manual usage, a file download may open a system popup (Save As window) asking where to save the file.
* Browser automation tools like Playwright and Selenium cannot directly handle OS-level popups reliably.
* Playwright avoids this problem by capturing the download event and allowing the file to be saved programmatically.

Easy Remember:

* Click Download ➜ Download Event Triggered ➜ Playwright Captures File ➜ Save File Automatically
* No need to handle the system "Save As" popup.

eg.

package org.sujalkhot.day18;

import java.nio.file.Path;
import java.nio.file.Paths;

import com.microsoft.playwright.Download;
import com.microsoft.playwright.Page;

public class DownloadFile {

	public static void main(String[] args) {

		Page page = BrowserFactory.initBroswer("chrome", false, true);
		page.navigate("https://rahulshettyacademy.com/upload-download-test");

		// Download File
		Download download = page.waitForDownload(() -> {
			page.locator("text=Download").click();
		});

		// save file
		Path downloadedFile = Paths.get("downloads", download.suggestedFilename());
		download.saveAs(downloadedFile);

		System.out.println("Downloaded File to : " + downloadedFile.toAbsolutePath());
		//Absolute path give exact path of our file
		//Downloaded File to : C:\Users\sujal\OneDrive\Documents\PlayWright Project\PlayWright-Project\playwright\downloads\download.xlsx

	}

}

- Search boxes

==================================================
How to Handle Auto-Suggest Dropdown
==================================================

Steps:

1. Type text
2. Wait for suggestions
3. Select the required suggestion

--------------------------------------------------

Example

page.fill("#search", "jav");

Locator suggestions =
    page.locator(".suggestions div");

suggestions
    .filter(
        new Locator.FilterOptions()
            .setHasText("JavaScript")
    )
    .first()
    .click();

- Types "jav"
- Finds matching suggestion
- Clicks "JavaScript"
- .first().click() because if 2 JAVASCRIPT is present it throw an error

--------------------------------------------------

Loop Through Suggestions

Locator suggestions =
    page.locator(".suggestions div");

for (int i = 0; i < suggestions.count(); i++) {

    String text =
        suggestions.nth(i).innerText();

    if (text.equals("Java")) {

        suggestions.nth(i).click();
        break;
    }
}

- Reads each suggestion
- Compares text
- Clicks matching value

public void selectFromAutoSuggest(
    String inputBoxLocator,
    String inputText,

Locator in Playwright (Java)

Important Concept:

- A Locator can represent one element or multiple elements.
- Playwright does not have findElement() or findElements() methods like Selenium.
- Playwright uses the Locator API.

==================================================
Selenium - findElement()
==================================================

Returns a single element.

Example:

WebElement username =
    driver.findElement(
        By.id("username")
    );

username.sendKeys("venkat");

Behavior:

- Returns one WebElement.
- Throws NoSuchElementException if element is not found.

==================================================
Playwright - locator()
==================================================

Returns a Locator.

Row 0 usually contains headers.

Example:

Row 1 → admin | admin123

Starting from 1 skips the header row.

   ↓

   ↓

   ↓

   ↓

Interview Answer
==================================================

To read all rows in Apache POI, first get the sheet using getSheet(), retrieve the last row number using getLastRowNum(), and then loop through the rows using getRow(). Cell values can be accessed using getCell() and converted to strings using toString(). Header rows are usually skipped by starting the loop from row index 1.

TestNG - Retry Analyzer (Auto Re-Run Failed Tests)

==================================================

==================================================

✗ Manual process

✗ Need to run failed suite again

--------------------------------------------------

✓ Automatic retry

✓ No manual intervention

✓ Useful for flaky tests

==================================================
What is IRetryAnalyzer?
==================================================

IRetryAnalyzer is a TestNG interface used to automatically re-execute failed test cases.

If a test fails:

TestNG calls retry()

and decides whether to run the test again.

==================================================
RetryAnalyzer Example
==================================================

public class RetryAnalyzer
        implements IRetryAnalyzer {

    int counter = 0;

    int retryLimit = 1;

    @Override
    public boolean retry(
            ITestResult result) {

        if (counter < retryLimit) {

            counter++;

            return true;
        }

        return false;
    }
}

==================================================
How It Works
==================================================

retryLimit = 1

Execution:

Attempt 1
→ FAIL

Attempt 2
→ RETRY

If passes
→ PASS

If fails again
→ FAIL

==================================================
Execution Example
==================================================

retryLimit = 2

Attempt 1 → FAIL

Attempt 2 → RETRY

Attempt 3 → RETRY

Still fails

↓

Final Result = FAIL

==================================================

==================================================

IAnnotationTransformer automatically attaches RetryAnalyzer to all test methods.

No need to add:

@Test(retryAnalyzer = RetryAnalyzer.class)

for every test.

public class AnnotationTransformer
        implements IAnnotationTransformer {

    public void transform(
            ITestAnnotation annotation,
            Class testClass,
            Constructor testConstructor,
            Method testMethod) {

        annotation.setRetryAnalyzer(
                RetryAnalyzer.class);
    }
}

- Mainly used in Selenium, Playwright, and Appium automation.

==================================================
What is a Thread?
==================================================

- A thread is a lightweight process.
- Each thread executes a test independently.

Example:

Thread 1 → Login Test

Thread 2 → Search Test

Thread 3 → Checkout Test

==================================================
What is thread-count?
==================================================

- Defines the maximum number of threads TestNG can create.
- More threads = More tests running simultaneously.

Example:

thread-count="3"

Means:

Maximum 3 tests can run at the same time.

==================================================
What is parallel Attribute?
==================================================

Defines what should run in parallel.

Types:

1. methods
2. classes
3. tests
4. instances

==================================================
parallel="methods"
==================================================

Runs test methods in parallel.

Methods of the same class can run simultaneously.

Example:

@Test
public void testA() {}

@Test
public void testB() {}

@Test
public void testC() {}

Execution:

Thread 1 → testA

Thread 2 → testB

Thread 3 → testC

Use Case:

✓ Independent test methods

==================================================
parallel="classes"
==================================================

Runs classes in parallel.

Methods inside each class run sequentially.

Example:

LoginTest

SearchTest

Execution:

Thread 1 → LoginTest

Thread 2 → SearchTest

Use Case:

✓ Multiple test classes

==================================================
parallel="tests"
==================================================

Runs <test> tags in parallel.

Most commonly used for Cross Browser Testing.

Example:

<test name="Chrome">

<test name="Firefox">

Execution:

Thread 1 → Chrome

Thread 2 → Firefox

Use Case:

✓ Cross Browser Testing

parallel="instances"
==================================================

Runs object instances in parallel.

Usually used with:

package utils;

import org.testng.IRetryAnalyzer;
import org.testng.ITestResult;

public class RetryAnalyzer implements IRetryAnalyzer {

	int counter = 0;
	int retryLimit = 1;

	@Override
	public boolean retry(ITestResult result) {
		if (counter < retryLimit) {
			counter++;
			return true;
		}
		return false;
	}

}

TestNG Assertions

==================================================
What are Assertions in TestNG?
==================================================

Assertions are validation points in a test.

They compare:

Expected Result

vs

Actual Result

and decide whether a test should PASS or FAIL.

Without assertions, TestNG cannot verify if the application behaved correctly.

==================================================
Types of Assertions
==================================================

1. Hard Assertions
   (org.testng.Assert)

2. Soft Assertions
   (org.testng.asserts.SoftAssert)

==================================================
1. Hard Assertions
==================================================

- Provided by org.testng.Assert
- If an assertion fails, test execution stops immediately.
- Remaining statements are NOT executed.

==================================================
Example
==================================================

@Test
public void loginTest() {

    String actualTitle = "Dashboard";

    Assert.assertEquals(
        actualTitle,
        "Dashboard"
    );

    System.out.println(
        "This will execute"
    );

    Assert.assertTrue(
        actualTitle.contains("Dash")
    );

    System.out.println(
        "This will also execute"
    );
}

==================================================
If Assertion Fails
==================================================

@Test
public void loginTest() {

    String actualTitle = "Home";

    Assert.assertEquals(
        actualTitle,
        "Dashboard"
    );

    System.out.println(
        "This line will NOT execute"
    );
}

Result:

✗ Test Failed

Execution stops immediately.

==================================================
Common Hard Assertions
==================================================

Assert.assertEquals(actual, expected);

→ Compare values

--------------------------------------------------

Assert.assertNotEquals(a, b);

→ Ensure values are different

--------------------------------------------------

→ Condition must be true

--------------------------------------------------

Assert.assertFalse(condition);

→ Condition must be false

--------------------------------------------------

→ Object should be null

--------------------------------------------------

→ Object should not be null

--------------------------------------------------

Assert.fail();

→ Forcefully fail a test

==================================================
Examples

✓ PASS

--------------------------------------------------

✓ PASS

--------------------------------------------------

✓ PASS

--------------------------------------------------

✓ PASS

Interview Answer

Hard Assertions in TestNG are provided by the Assert class. If an assertion fails, the current test method stops execution immediately and the remaining steps are skipped. They are used to validate expected and actual results in automation tests.

TestNG Soft Assertions

==================================================
What are Soft Assertions?
==================================================

Soft Assertions are provided by:

org.testng.asserts.SoftAssert

Unlike Hard Assertions:

✓ Test execution continues even if an assertion fails.

✓ All assertion failures are collected.

✓ Failures are reported only when assertAll() is executed.

==================================================
Why Use Soft Assertions?
==================================================

Used when you want to validate multiple UI elements in a single test.

Examples:

✓ Page Title

✓ Logo

✓ Username

✓ Welcome Message

✓ Buttons

Instead of stopping at the first failure, SoftAssert validates everything and reports all failures together.

==================================================
Example
==================================================

@Test
public void softAssertMethodsDemo() {

    SoftAssert sa = new SoftAssert();

    sa.assertEquals("Login", "Login");

    sa.assertNotEquals(10, 20);

    sa.assertTrue(5 > 2);

    sa.assertFalse(5 < 2);

    sa.assertNotNull("TestNG");

    sa.assertNull(null);

    sa.assertAll();   // Mandatory
}

==================================================
Important Rule
==================================================

assertAll() is mandatory.

Without assertAll():

✓ Test will appear PASS

✓ Failures will be ignored

==================================================
Wrong Example
==================================================

SoftAssert sa = new SoftAssert();

sa.assertEquals("Login", "Dashboard");

// Missing assertAll()

Result:

✓ TestNG shows PASS

✗ Failure is ignored

==================================================
Correct Example
==================================================

SoftAssert sa = new SoftAssert();

sa.assertEquals("Login", "Dashboard");

sa.assertAll();

Result:

✗ Test FAILS

==================================================
Common Soft Assertion Methods
==================================================

sa.assertEquals(actual, expected);

→ Checks equality

--------------------------------------------------

→ Checks inequality

--------------------------------------------------

→ Condition must be true

--------------------------------------------------

→ Condition must be false

--------------------------------------------------

sa.assertNull(object);

→ Object should be null

--------------------------------------------------

sa.assertNotNull(object);

→ Object should not be null

--------------------------------------------------

sa.assertSame(actual, expected);

→ Same object reference

--------------------------------------------------

sa.assertNotSame(actual, expected);

→ Different object reference

--------------------------------------------------

sa.fail();

→ Forcefully fail test

--------------------------------------------------

sa.assertAll();

→ Reports all failures (MANDATORY)

Framework Notes (Easy Interview Notes)

==================================================
@BeforeClass vs @BeforeMethod
==================================================

@BeforeClass

- Used for Browser level setup.
- Executes only once before all test methods in a class.
- Browser launch is usually done here.

Example:

@BeforeClass
public void setup() {
    browser = playwright.chromium().launch();
}

--------------------------------------------------

@BeforeMethod

- Used for Page level setup.
- Executes before every test method.
- New BrowserContext and Page are usually created here.

Example:

@BeforeMethod
public void createPage() {
    context = browser.newContext();
    page = context.newPage();
}

==================================================
alwaysRun = true
==================================================

- Forces a configuration method to execute even if previous methods fail.
- Commonly used with @AfterMethod and @AfterSuite.

Example:

@AfterMethod(alwaysRun = true)
public void tearDown() {
    browser.close();
}

Benefits:

✓ Cleanup always happens
✓ Browser closes properly
✓ Reports get generated
✓ Resource leaks avoided

Interview Answer
==================================================

In a Playwright POM framework, Browser is usually initialized in @BeforeClass because it is browser-level setup. BrowserContext and Page are created in @BeforeMethod because they are test-level resources. Configuration values such as browser, baseUrl, headless mode, retry count, and report directory are stored in config.properties. Page objects are passed through constructors so that locators and actions can use the same Playwright Page instance across the framework.

Interview Answer
==================================================

Cucumber is a BDD (Behavior Driven Development) framework that allows automation tests to be written in plain English using Gherkin language. It uses keywords such as Given, When, Then, And, and But. Feature files contain business-readable scenarios, while Step Definition classes contain the Java code that automates those scenarios.

- if you run directyly cucumber file ie. Test.feature 
- in Console they give you direct code to implements 
- copy that code paste it in class and give implementation

| Plugin                                       | Output      | Use                                                                                   |
| -------------------------------------------- | ----------- | ------------------------------------------------------------------------------------- |
| `pretty`                                     | Console     | Displays formatted, readable test execution in the terminal.                          |
| `html:target/cucumber-reports/cucumber.html` | HTML Report | Generates a browser-friendly report of test results.                                  |
| `json:target/cucumber-reports/cucumber.json` | JSON Report | Creates a machine-readable report for reporting tools (e.g., Allure, Extent Reports). |
| `junit:target/cucumber-reports/cucumber.xml` | XML Report  | Generates a JUnit XML report for CI/CD tools like Jenkins and GitHub Actions.         |
